### Phân đoạn 1.1: Bài toán dữ liệu đang được sử dụng - Data in Use (00:00 - 05:00)
- **Lời thoại AI (Audio Voiceover):**
  "Welcome to Act One. How can a client use cloud computation without revealing private data? The roadmap moves from plain values to encrypted objects. T-L-S protects information in transit, but an ordinary server decrypts it before computing. Homomorphic Encryption keeps the Secret Key with the client and allows only ciphertext to cross the trust boundary.

  The first experiment shows the outsourced-compute dilemma. A private image begins with the data owner and moves toward useful cloud hardware. The transfer itself may be protected, yet the server eventually places a readable representation in memory. The scanning line and observing eye make the missing security state visible: the data is exposed while the computation is active. Encryption at rest protects a stored file, and encryption in transit protects network packets, but neither one automatically protects server memory. Homomorphic Encryption targets this third state, called data in use. The goal is not to hide the existence of a request. The goal is to prevent the cloud from learning the input, intermediate features, or final numerical result while still carrying out the requested arithmetic.

  Now define the two trust domains precisely. The trusted client owns the private input and the Secret Key. The cloud is allowed to receive ciphertext and an evaluation program, but it is not given decryption authority. Watch the encrypted packet cross the dashed boundary while the key remains on the client side. The crossed-out eye means that the cloud can store and transform the ciphertext without reading its hidden values. This separation is stronger than simply calling one machine secure and another machine insecure. It assigns a specific capability to each object. The client may encrypt and decrypt. The server may evaluate the approved circuit. Only the client may reveal the answer. A clear trust boundary is therefore the starting point for both the cryptographic design and the deployment architecture.

  The tunnel on screen represents transport encryption such as T-L-S. The packet remains protected while it moves through the network. At the server endpoint, however, the tunnel ends. A conventional application decrypts the request, reconstructs the plaintext image, and places that image in memory so ordinary code can inspect and process it. This is why a secure channel is not the same thing as encrypted computation. T-L-S can prevent an outside observer from reading packets in transit, but it does not prevent the receiving service, a compromised process, or a memory inspection tool from seeing the decoded data. The visual change from ciphertext dots back into a readable grid marks that exact boundary. Protecting the route is necessary, but the computation itself requires a different mechanism.

  Run the same request again using Homomorphic Encryption. The client begins with a readable value, encrypts it into an opaque ciphertext, and sends only that ciphertext to the evaluator. The cloud applies the function without opening the hidden message. Notice that the object remains encrypted while it enters the evaluator, while the evaluator turns, and while the result travels back across the boundary. The returned object is Enc of f of x, not the plaintext answer. Only after it reaches the client and meets the Secret Key does it become the visible score. This preserves privacy for the input, the intermediate encrypted state, and the output. The essential promise is simple: useful computation occurs on the cloud, but decryption authority never leaves the trusted client.

  Stronger privacy has a measurable cost. Six small plaintext values expand into a much larger ciphertext representation, so storage and communication increase. The two operation paths show a second limitation. Algebraic addition is directly supported, while a hidden branch such as if x is positive cannot be evaluated as ordinary control flow. The clock then shows the third cost: operations on encrypted polynomials require far more work than operations on native numbers. These constraints interact. Larger parameters may provide more security, precision, or multiplication depth, but they also increase memory use and runtime. A practical FHE system therefore chooses the value layout, supported operations, parameter sizes, and circuit depth together. Homomorphic Encryption is not free encryption around an unchanged program. It is a deliberate systems tradeoff."

### Phân đoạn 1.2: Tính đồng cấu và mô hình mạch số học (05:00 - 10:00)
- **Lời thoại AI (Audio Voiceover):**
  "The second chapter asks whether encrypted computation produces the correct numerical answer. We first follow x through encryption and evaluation, then compare exact integer schemes with approximate C-K-K-S, and finally expose multiplication depth as a finite circuit resource.

  Correctness is the central invariant. The upper lane performs ordinary plaintext computation. It sends the value two through f of x equals x squared plus one and obtains five. The lower lane starts from an encrypted version of the same value. The cloud applies Evaluate f without learning the input and produces another locked object. When the client decrypts that result, the lower lane also reveals five. In an exact scheme the answers should agree exactly within the message space. In C-K-K-S the decrypted value should be numerically close, because the scheme intentionally uses approximate arithmetic. The important comparison is between the two complete paths: decrypt after encrypted evaluation should match ordinary evaluation within the permitted error. If those paths disagree, privacy alone is not enough; the computation is not useful.

  Encrypted circuits also have a resource budget. The stacked levels represent the active coefficient-modulus chain. Each time the locked token passes through a multiplication, it moves downward and consumes part of the available depth. Additions are usually cheaper and do not spend a level in the same way. A leveled scheme chooses enough parameters for a known multiplication schedule, so the circuit must be planned before private data is evaluated. When the token approaches the bottom, bootstrapping offers a way to refresh it and return to a higher-capacity state. That capability can support longer computations, but the large bootstrap gate and its warning label are intentional: refresh is powerful and expensive. Good circuit design tries to control depth, scale, and precision before relying on a costly reset.

  Ordinary programs often branch after inspecting a value. The encrypted token cannot do that, because the cloud does not know whether the hidden number is positive, larger than another number, or equal to a threshold. On screen, the token reaches the red if statement and stops at the wall. The replacement path is an arithmetic circuit whose structure is fixed in advance. Addition and multiplication gates do not need to inspect a secret condition; they apply algebraic transformations to every ciphertext that reaches them. This changes the programming model. Comparisons, maximum functions, secret-dependent indexing, and loops with hidden stopping conditions must be removed, approximated, or implemented through more specialized protocols. For C-K-K-S workloads, program design moves away from control flow and toward polynomial-compatible arithmetic.

  Private inference does not require every object to be encrypted. The red feature map represents private client data and must remain ciphertext. The blue column represents public model weights that the server already owns. Both flow into the multiplication gate, where a ciphertext-plaintext product applies the public weights to the hidden features. The output remains encrypted, and the crossed-out eye confirms that the server still cannot inspect the resulting activation. This arrangement is useful because multiplying by encoded plaintext constants is often cheaper than multiplying two ciphertexts. It also matches a common deployment model: the service provider may expose or protect its model under a separate policy, while the user requires confidentiality for inputs, intermediate activations, and outputs. The cryptographic boundary follows data ownership, not a rule that everything must look identical.

  The message type determines the scheme. Exact integers such as seven, twelve, and four move into the B-F-V or B-G-V region, where arithmetic is performed over modular integer messages. Decimal values such as zero point two five, negative one point four, and three point one two move into C-K-K-S. C-K-K-S accepts controlled approximation so that real or complex numerical workloads can be represented efficiently. The lower grid then shows its major performance feature: many values can occupy S-I-M-D slots inside one encoded polynomial and one ciphertext. A single homomorphic instruction can act across those slots in parallel. Computer vision relies heavily on real-valued tensors and repeated numerical operations, so approximate packed arithmetic is usually the better fit for this course. The approximation is managed, measured, and kept within an application tolerance."

### Phân đoạn 1.3: Vành đa thức, RLWE và trực giác về nhiễu (10:00 - 15:00)
- **Lời thoại AI (Audio Voiceover):**
  "The final chapter opens the ciphertext. A real vector becomes a ring polynomial; encryption adds error; the result is a noisy polynomial tuple with finite resources.

  Why do polynomials appear? The four numerical values on the left begin as an ordinary vector. Each value moves to a coefficient position on the polynomial axes, where the stems make the new representation visible. The quotient-ring circle then gathers that coefficient structure into the bounded algebraic workspace used by the scheme. Microsoft S-E-A-L stores plaintexts and ciphertext components as polynomials whose coefficients are reduced modulo an integer q. Addition combines coefficients efficiently, and multiplication uses fast polynomial arithmetic. The ring is not decoration around a hidden scalar. It is the actual language in which encoding, encryption, and homomorphic evaluation are expressed. This representation gives the library a finite machine object while preserving the algebra needed for valid encrypted operations. The next experiment explains how the quotient prevents polynomial degree from growing without bound.

  The circular path represents the quotient relation. We work in Z sub q of X modulo X to the N plus one, so X to the N is identified with negative one. The token begins as the high-degree term X to the N plus two. As it travels around the circle and crosses the X to the N boundary, its sign changes. It then lands at the X squared position and exits as negative X squared. This is negacyclic wrap-around: powers do not continue growing forever, and wrapping across degree N introduces a minus sign. Only a few labeled positions are needed to see the rule. The quotient keeps every result below degree N, which makes repeated polynomial addition and multiplication remain inside one fixed-dimensional ring.

  The R-L-W-E picture combines three ingredients. The green row is a secret polynomial with small coefficients. The blue row supplies randomized public information, and the grey dots represent a small error term. All three enter the mixer, which produces two apparently noisy ciphertext polynomials. The error is not an accidental defect added after encryption. It is essential to the Ring Learning With Errors security assumption because it hides the exact algebraic relationship involving the secret. Without the Secret Key, recovering that relationship from many noisy samples is believed to be computationally hard for appropriate parameters. At the same time, the error cannot be arbitrary. It must remain small enough that legitimate decryption and decoding still recover the intended approximate message. Security and correctness therefore depend on the same carefully controlled noisy structure.

  There is no single counter that fully describes C-K-K-S capacity. The three bars track related resources: coefficient modulus, remaining levels, and meaningful numerical precision. The locked token begins with a small error halo. After each multiplication, the active modulus and level capacity shrink while approximation effects accumulate. The halo expands and moves closer to the red failure threshold. Fresh encryption needs randomized error for security, but evaluation, rescaling, and finite-precision encoding also influence the final numerical result. Parameters must leave enough headroom for the complete circuit, not merely for one operation. Correct decoding requires compatible scales, compatible parameter levels, sufficient modulus, and enough precision for the application tolerance. It is therefore more accurate to plan a combined modulus, level, scale, and precision budget than to speak about one generic noise number.

  The final transformation connects this foundation to the next scene. A real-valued vector first enters a set of logical C-K-K-S slots. The encoder maps those slot values into a polynomial representation and records a scale that preserves useful fractional precision. Encryption then turns the encoded plaintext into a tuple of noisy ring polynomials. The evaluator receives that ciphertext and applies only operations supported by the planned arithmetic circuit. Throughout the path, the visible numerical meaning and the polynomial object are different layers of the same computation. The client reasons about vectors and approximate results, while the cryptographic library manipulates coefficients, moduli, levels, and keys. Scene 2 will open the encoder in detail and show how slot packing, scale, rotations, and rescaling make this representation practical for numerical workloads."
