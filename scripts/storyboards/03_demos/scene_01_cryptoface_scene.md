### Phân đoạn 1.2: Kiến trúc CryptoFaceNet & Tối ưu CKKS (03:30 - 07:30)
- **Lời thoại AI (Audio Voiceover):**
  "To successfully integrate Fully Homomorphic Encryption with Deep Learning, we must immediately confront the most formidable barrier in modern cryptography: Multiplicative Depth. Look at the challenge currently presented on the screen. In standard computer vision tasks, achieving high accuracy in facial recognition typically requires extremely deep Convolutional Neural Networks, stacking dozens of layers as seen in standard ResNet or VGG architectures. However, the CKKS homomorphic encryption scheme operates under the 'Learning With Errors' mathematical assumption. This means every ciphertext inherently contains a foundational layer of cryptographic noise. While homomorphic addition scales this noise linearly and safely, homomorphic multiplication—which is strictly required for every convolutional filter and polynomial activation function—causes this noise to grow exponentially.

Watch the diagram. If we force an encrypted image through a standard, deep CNN, the network performs thousands of sequential multiplications. As the ciphertext progresses through these deep layers, the noise accumulation becomes catastrophic. Long before the network reaches its final output layer, the designated noise budget is completely exhausted. The mathematical error overtakes the actual encoded biometric data, leading to what we define as 'CKKS Data Collapse'. The resulting ciphertext is permanently corrupted, decrypting into entirely random, unintelligible garbage. Therefore, applying traditional deep learning architectures directly to encrypted data is mathematically impossible.

As system architects, we cannot alter the fundamental laws of lattice cryptography to accommodate deep networks. Instead, we must fundamentally redesign the neural network architecture to accommodate the cryptography.Notice the transition on the screen to the CryptoFaceNet solution. Instead of feeding the entire high-resolution facial image into a single, massive neural network, the system employs a 'Patch-based Processing' strategy.The input image is algorithmically divided into a grid of discrete, non-overlapping squares, represented here by the $4 \times 4$ patch grid. Each of these localized patches contains specific facial textures—such as the curve of an eyebrow, the geometric edge of a jawline, or the micro-texture of the iris. By breaking the global image down into these independent local components, we drastically reduce the complexity of the feature extraction task required for any single section of the encrypted tensor.

With the facial data fragmented into patches, we can now leverage the computational supremacy of the CKKS scheme: the Single Instruction, Multiple Data, or SIMD property. Watch as the patches are distributed across the system. Instead of processing the entire face through a deep sequential pipeline, each encrypted patch is routed into a highly specialized, incredibly shallow neural network known as a Patch CNN, or PCNN. Because these PCNNs are intentionally shallow—consisting of only the absolute minimum number of convolutional layers necessary to extract local features—they require an extremely low multiplicative depth. By executing these shallow networks in parallel across all patches simultaneously, we achieve two monumental victories. First, we completely prevent the catastrophic noise accumulation, ensuring the ciphertext remains fully decryptable. Second, we drastically reduce the computational latency on the cloud server.

Having solved the neural network bottleneck, we can now map out the complete, end-to-end lifecycle of the CryptoFace system. On screen, we are establishing a strict Zero-Trust architectural layout. The vertical divider represents the network boundary: separating the trusted, local 'Client' environment on the left, from the untrusted 'Server' or Cloud environment on the right. Furthermore, the horizontal divider separates the two primary chronological phases of biometric security. The top quadrants will handle the 'Enrollment' phase, where a user registers their identity for the very first time. The bottom quadrants will handle the 'Verification' phase, where a user attempts to authenticate themselves months or years later. Let us trace the exact cryptographic flow of data through these quadrants."

### Phân đoạn 1.3: Luồng Đăng ký - Enrollment (07:30 - 11:30)
- **Lời thoại AI (Audio Voiceover):**
  "Let us now direct our attention to the top half of our architectural layout, which represents the Enrollment phase. In any biometric security framework, the lifecycle must begin by establishing a foundational ground truth—a reference profile for the user.Operating strictly within the secure confines of the trusted Client environment—such as the user's personal smartphone or a highly secure registration terminal—the system initiates the capture process. As you can see, a high-resolution 'Reference Face' is captured by the camera. Simultaneously, a plaintext 'Identity ID', such as an employee number or a username, is generated to tag this session. In a traditional, legacy system, this highly sensitive, raw biometric photograph would simply be transmitted directly to the central server for processing. But in the Zero-Trust CryptoFace architecture, this raw data is strictly prohibited from leaving the client device.

Instead of transmitting the image, the client device invokes a local cryptographic module. Notice the encryption box appearing on the screen. The device generates an asymmetric cryptographic key pair based on the Ring Learning With Errors assumption. It utilizes the Public Key to mathematically lock the raw image. During this encryption process, the pixel intensity values of the facial image are mapped into high-dimensional polynomial rings, and a carefully calibrated layer of mathematical noise is injected to completely obfuscate the underlying data. The output of this operation is the Ciphertext you see now. The plaintext face has been completely destroyed and replaced by an impenetrable cryptographic vault. Because the client used a Public Key, the data is securely locked, but the corresponding Secret Key—the only mathematical tool capable of reversing this process—remains safely stored on the local hardware. The ciphertext now securely crosses the network boundary to the untrusted server.

Now we witness the most groundbreaking capability of Fully Homomorphic Encryption: Homomorphic Feature Extraction. The untrusted Cloud Server receives the encrypted facial image. Watch as the ciphertext is fed directly into the server's Patch Convolutional Neural Network. Remember, the server does not hold the Secret Key, so it cannot see the face. However, by performing homomorphic additions and multiplications on the polynomial structures, the neural network essentially computes blindly. The convolutional filters slide across the encrypted data, successfully identifying edges, contours, and complex facial patterns without ever revealing the image itself. The output of this neural network is a highly compressed, encrypted Reference Vector—an embedding that captures the absolute biometric essence of the user. The server has successfully executed complex artificial intelligence on pure cryptographic noise.

Finally, we reach the conclusion of the Enrollment phase: Blind Storage. The cloud server takes the newly generated, highly compressed Encrypted Reference Vector and stores it securely inside its centralized Database. To ensure this encrypted biometric profile can be retrieved in the future, the server maps it to the plaintext Identity ID that was provided by the client at the beginning of the session. The user is now fully enrolled in the system. The beauty of this architecture is absolute privacy. If a malicious hacker were to breach the server's database right now, or if a rogue database administrator attempted to view the records, they would not find any faces. They would only find random, unintelligible polynomial coefficients. The baseline security has been mathematically guaranteed."

### Phân đoạn 1.4: Luồng Xác thực - Verification (11:30 - 15:00)
- **Lời thoại AI (Audio Voiceover):**
  "We now move to the bottom half of our architectural diagram, representing the Verification phase. Imagine it is now months, or even years, after the initial enrollment. The user is attempting to access a highly secure facility or unlock a digital vault. Operating once again within the trusted Client environment, the local camera captures a fresh, live biometric reading, known as the 'Probe Face'. Simultaneously, the user provides a 'Claimed ID'—asserting who they are supposed to be. Just as in the enrollment phase, the zero-trust protocol mandates that this live biometric data cannot be exposed. The client device immediately invokes the cryptographic module. It applies the same Public Key to the live probe face, injecting fresh mathematical noise, and generating a brand new ciphertext. This securely locked probe face is now transmitted across the network to the untrusted cloud server. From the moment the camera snapped the picture, the data was secured.

Upon receiving the data, the cloud server initiates two parallel operations. First, it reads the plaintext Claimed ID. It uses this identifier to query its centralized database and retrieve the user's original, encrypted 'Reference Vector'—the exact same ciphertext that was blindly stored during the enrollment phase months ago. Simultaneously, the server routes the newly arrived, encrypted Probe Face into the Patch-based Convolutional Neural Network. Once again, the network computes blindly. It performs homomorphic feature extraction on the live biometric data, eventually outputting a brand new, encrypted 'Probe Vector'. The server now holds two distinct, highly compressed encrypted arrays: the reference vector from the past, and the probe vector from the present. The challenge now is to determine if these two vectors belong to the same human being, without ever decrypting them.

To determine the similarity, the server performs a Secure Comparison. It homomorphically calculates the distance between the two vectors, outputting an encrypted 'Score'. In traditional programming, verifying this score is trivial. The server would simply use a conditional 'If' statement: If the score is less than the security threshold, grant access. However, Fully Homomorphic Encryption fundamentally prohibits conditional branching. The server cannot evaluate an 'If' statement because it cannot see the numbers. It cannot know if the score is smaller or larger than the threshold. To bypass this, the server performs one final homomorphic arithmetic operation: it mathematically subtracts the predefined security threshold from the encrypted score. The formula 'result equals score minus threshold' is executed entirely in the encrypted domain. The output is a highly secured, locked Result Box. The server has completed the authentication logic, yet it remains completely unaware of whether the verification was successful or if it failed.

Because the server is mathematically blind to the outcome, it must return this locked result box to the trusted Client. As the ciphertext crosses back over the zero-trust boundary, the client device receives the payload. It applies the exclusive Secret Key—which has never left the local hardware—to the decryption module. The complex lattice noise is instantly stripped away. The client device evaluates the underlying plaintext result. Because the original homomorphic score was lower than the threshold, the plaintext result is a negative mathematical value. The local client recognizes this negative value as a definitive cryptographic match. Access is Granted. The biometric authentication is complete. The user has successfully verified their identity utilizing the massive computational power of the cloud, while mathematically guaranteeing that neither their facial data, nor the outcome of the authentication, was ever exposed to the server or the network. This is the ultimate promise of CryptoFace."
