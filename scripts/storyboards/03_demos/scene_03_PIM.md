Now, we are scaling our cryptographic ambition to a much broader and more challenging problem: Secure Database Retrieval.

Let us first examine the fatal flaw in how traditional database systems operate today. Currently, to perform a search, the client must transmit their query image to the centralized server.We heavily rely on Transport Layer Security, or TLS, to encrypt this data while it travels across the network. This protects the image from external hackers or man-in-the-middle attacks during transit. Furthermore, the database itself might be encrypted at rest. However, there is a fundamental cryptographic gap when the data arrives at its destination.
  
  The moment the server needs to execute the search algorithm, it must decrypt the query image into plaintext inside its system memory. This exposes the system to a critical vulnerability known as 'Query Leakage'.Even if we completely trust the server to not be hacked by outsiders, we must consider the 'Honest-but-Curious' threat model. An insider threat, a curious database administrator, or a compromised server log can instantly see the exact image the investigator is looking for. In high-stakes scenarios, the intent of the search is just as sensitive as the data itself. If a hospital queries a DNA sequence for a rare genetic disorder, or an intelligence agency searches for a specific political figure, exposing the query immediately compromises the entire investigation. The server learns who you are targeting, when you are targeting them, and what your motives are. 
  
  This specific vulnerability—the exposure of the client's intent—is exactly what Private Image Matching is engineered to eradicate.By deploying Fully Homomorphic Encryption, we transition from standard database retrieval to Zero-Knowledge Querying. In this advanced paradigm, the client mathematically locks their query image before it ever leaves their local device. The server will be forced to compute the complex visual matching algorithms directly on the ciphertext. It will scan its entire database and return the matching result, while remaining completely and mathematically blind to what the investigator is actually searching for. Let us see how this cryptographic process unfolds.

### Phân đoạn 3.2: Tiền xử lý & Đóng gói Đặc trưng (03:30 - 07:30)
- **Lời thoại AI (Audio Voiceover):**
  "To execute this zero-knowledge query efficiently, we must address a monumental computational bottleneck. A standard high-resolution image consists of millions of individual pixels. If we were to naively encrypt the raw image, applying the Fully Homomorphic Encryption algorithm to every single pixel color channel, the resulting ciphertext would explode in size. It would generate gigabytes of encrypted data for just a single image. Transmitting this across a network, let alone computing deep learning convolutions on it at the server, is entirely impractical.Therefore, HERS and Private Image Matching adopt an Edge Computing architecture. The preprocessing begins entirely locally at the client’s secure environment. As you can see, the raw query image is passed through a lightweight, pre-trained Convolutional Neural Network deployed directly on the investigator's device.
  
  This local CNN acts as a highly optimized feature extractor. Instead of outputting an image, it distills the complex facial structures into a dense, one-dimensional Feature Vector—often referred to as an embedding.Look at the vector now generated on screen. This array of floating-point numbers—perhaps 256 or 512 dimensions—captures the absolute semantic identity of the subject. This step inherently adds a foundational layer of privacy, as a feature vector cannot be easily reverse-engineered back into a human face. However, to achieve true cryptographic guarantees against an untrusted national database, this plaintext vector is still not secure enough. It must be encrypted before it touches the internet.
  
  This brings us to a brilliant optimization in modern lattice-based cryptography. If we encrypted each of the 512 floating-point numbers into 512 separate ciphertexts, we would still face a massive network overhead. To solve this, we utilize the SIMD—Single Instruction, Multiple Data—capability intrinsic to the CKKS homomorphic scheme. Notice the 'Plaintext Block' with 4096 slots. In the CKKS scheme, a single plaintext object is represented by a high-degree polynomial. We can mathematically pack our entire 512-dimensional feature vector into the independent coefficients, or 'message slots', of this single polynomial. Watch as the vertical vector rotates and is smoothly compressed into the plaintext slots. This batching technique means that any homomorphic operation the server later applies to this ciphertext will be executed on all 512 features simultaneously, providing a massive degree of parallel computation.
  
  Finally, we reach the encryption phase. The client applies their specific Public Key to this densely packed plaintext polynomial. Through the Ring Learning With Errors mathematical assumption, highly calibrated noise is injected, completely obfuscating the data. The result is a single, highly compact Encrypted Query Vector. This ciphertext effectively locks the entire semantic identity of the subject into a mathematically robust vault. The client now transmits this single ciphertext across the network to the Central Database. Because of the local feature extraction and SIMD packing, the network payload is incredibly small—only a few megabytes. The server receives the package, completely blind to the numbers inside, perfectly setting the stage for homomorphic database retrieval."

### Phân đoạn 3.3: Tính toán Khoảng cách Đồng hình (07:30 - 11:30)
- **Lời thoại AI (Audio Voiceover):**
  "Now that the securely encrypted query vector has reached the Central Database, we face the core challenge of Private Image Matching: The 1-to-N search over encrypted space. The server currently holds millions of reference vectors, represented here by the Database Grid. To identify the suspect or the medical match, the server must compute the similarity between our encrypted query and every single plaintext vector in its storage.In standard machine learning, we might use Cosine Similarity or Manhattan Distance. However, Fully Homomorphic Encryption strictly limits us to polynomial additions and multiplications. Calculating a square root or dividing by a vector norm in the encrypted domain is computationally devastating. Therefore, HERS relies on the Squared Euclidean Distance, represented by the formula on the screen: $D^2 = \sum (Q_i - V_i)^2$. This specific metric is chosen not just for its geometric accuracy, but because it is perfectly mathematically compatible with the limitations of the CKKS encryption scheme. We only need subtraction and multiplication to execute this.
  
  Let us break down this homomorphic evaluation step-by-step. First, the server must broadcast the single encrypted query vector to align with the massive scale of the database. Notice how the query is duplicated across all reference columns. The first arithmetic operation is Homomorphic Subtraction. The server subtracts the plaintext database vectors from our encrypted query tensor. In the realm of lattice-based cryptography, adding or subtracting a plaintext from a ciphertext is a relatively 'cheap' operation. It alters the underlying polynomial structure to represent the mathematical difference, but it causes almost zero expansion to the ciphertext size, and more importantly, it barely consumes any of our cryptographic noise budget. The result of this operation is a new set of ciphertexts, shown here as the 'Difference' blocks. At this exact millisecond, the server holds the precise numerical differences for every facial feature, yet because it is still wrapped in Ring Learning With Errors encryption, the server cannot read a single value.
  
  However, a simple difference includes negative values, which cancel out when summed. To calculate true geometric distance, we must square these differences. This is where we hit the most critical computational bottleneck in our architecture: Homomorphic Squaring. Watch the 'Noise' meter on the right closely as the squaring operation executes. Multiplying two ciphertexts together—even multiplying a ciphertext by itself—is extremely resource-intensive. It causes the underlying lattice dimensions to explode, requiring an expensive relinearization process to bring the ciphertext back to a manageable size. More critically, this multiplication causes the inherent mathematical error—the noise injected during the initial encryption at the client's device—to grow exponentially. You can see the noise budget immediately dropping and shifting to a warning yellow. This is the multiplicative depth penalty we discussed earlier. Every squared Euclidean computation brings the ciphertext one step closer to catastrophic data corruption.
  
  Despite the heavy consumption of the noise budget, the CKKS parameters were meticulously designed to withstand exactly this depth. The computation concludes successfully without breaching the noise threshold. The output of this entire parallelized operation is represented by the array of red blocks you see now: The Encrypted Distances. Think about the magnitude of what the server has just achieved. It has taken a high-dimensional encrypted vector, compared it geometrically against an entire database, and generated precise distance scores. Yet, from the server's perspective, it has merely manipulated high-entropy polynomials. It does not know which distance score is close to zero, and which is large. It cannot rank them, and it cannot filter them. The server possesses the exact answer to the investigator's query, but it is mathematically blinded from perceiving it."
